# 舵机 vs BLDC 关节电机：为什么需要不同的控制框架？

## 概述

本文档解释了为什么直接将基于 BLDC（无刷直驱电机）的控制框架应用于总线舵机会导致灾难性失败，以及 OpenServoSim 如何从底层解决这个问题。

## 核心差异对比

### 1. 控制模式

**BLDC 电机**：提供**力矩/电流透传**——控制器可以直接指定电机输出的力矩大小。这使得复杂的力位混合控制（如 MPC、RL policy）成为可能。

**总线舵机**：只接受**位置指令**——舵机内部有一个不可调节的 PID 控制器，外部只能告诉它"去到 X 度"。无法直接控制力矩。

### 2. 通信带宽

**BLDC 电机**：通过 EtherCAT 或 CAN 总线通信，延迟 < 1ms，控制频率可达 **1kHz+**。

**总线舵机**：通过半双工 UART 通信。一个 115200 baud 的串口，发送一帧指令约需 0.5ms，加上半双工切换延迟 2ms，10 个舵机逐个发送就需要 **25ms+**。实际控制频率只能做到 **50-100Hz**。

### 3. 响应特性

**BLDC 电机**：响应几乎瞬时，无死区，可以精确到 0.01°。

**总线舵机**：
- **死区**：内部 PID 通常有 ±1-2° 的死区，小于这个范围的指令变化会被忽略
- **延迟**：从收到指令到开始运动有 5-20ms 的延迟
- **滤波**：内部 PID 相当于一个低通滤波器，高频指令会被平滑掉

## 直接套用 BLDC 框架的后果

如果把 MIT Mini Cheetah 的控制算法（设计要求 1kHz 控制频率）直接用在 50Hz 的舵机上：

1. **控制器输出的高频力矩指令被舵机的内部 PID "吃掉"**
2. **通信延迟导致实际执行的是 25ms 前的指令**，相位滞后巨大
3. **控制器以为机器人在 A 状态，实际上已经在 B 状态**，疯狂振荡
4. **舵机过热、齿轮应力过大、结构件断裂**

## OpenServoSim 的解决方案

### 1. 仿真层"丑化"

在 MuJoCo 仿真中加入真实舵机的缺陷模型：
- 通信延迟模拟
- 死区模拟
- 低通滤波模拟

### 2. 算法层"降频"

使用 UVC 这种基于几何投射的低频算法（50Hz 即可），而非高频动力学算法。

### 3. 通信层"广播同步"

通过先写入 → 再广播触发的方式，确保所有舵机在同一瞬间开始运动。
